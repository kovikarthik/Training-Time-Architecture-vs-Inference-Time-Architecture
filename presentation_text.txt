Slide 3:
- 1. Training vs. Inference — The Core Distinction
- Training: optimize model weights to minimize loss
    Forward pass → compute predictions (large batch, B=4096)
    Backward pass → compute gradients via backpropagation
    Optimizer step → update weights (Adam: momentum + variance)
    Key: large GEMMs with high arithmetic intensity

Inference: use trained model to generate output tokens
    Prefill → process entire prompt (compute-bound, like training)
    Decode → generate tokens one-by-one (autoregressive)
    GEMMs collapse to GEMVs (matrix-vector multiplies)
    Key: memory-bandwidth-bound, not compute-bound
- Memory Footprint (Llama 3 8B)

Slide 4:
- 2. Problem Statement
- "Can a single hardware accelerator efficiently serveboth training and inference for large language models?"
- We answer this by implementing a config-driven analytical framework that:
    (a) Characterizes workloads via FLOP counting (6P training, 2P inference) and memory traffic estimation
    (b) Evaluates three architectures using a roofline-based bottleneck model
    (c) Computes energy-delay product (EDP) and 3-year total cost of ownership (TCO)
    (d) Quantifies the irreducible conflicts preventing a single chip from doing both efficiently

Slide 5:
- 3. Motivation — Economic and Technical Mismatch
- Inference dominates production compute:
    Training happens once; inference runs 24/7
    Inference accounts for 90%+ of compute hours
    Inference dominates total cost of ownership

Using training hardware for inference is wasteful:
    Our framework shows: H100 inference throughput
    is only 4,348 tok/s (memory-bound)
    Inference-ASIC achieves 19,471 tok/s (4.5× faster)
    at just 75W vs 700W (9.3× less power)

Specialized inference HW can save 45× on cost/token
- Mismatch: Every Dimension Opposes
- AI gap: 1,103 vs. 20.85 ops/byte → 53× difference

Slide 6:
- 4. Background — Roofline Performance Model
- Achievable Perf = min(Peak FLOP/s, AI × Bandwidth)

Arithmetic Intensity (AI) = FLOPs / Bytes transferred

Ridge Point = Peak FLOP/s / Bandwidth
    Above ridge point → compute-bound
    Below ridge point → memory-bound

Our framework (src/architecture/roofline.py) computes
AI for every (architecture, workload) pair and
determines the bottleneck automatically.
- Key Terms
- TFLOP/s: 10¹² floating-point ops per second
HBM3: High Bandwidth Memory (off-chip DRAM)
SRAM: On-chip static RAM (fast, expensive, limited)
EDP: Energy × Delay Product (lower = better)
TCO: Total Cost of Ownership (HW + electricity)
KV-cache: key/value tensors stored for
    autoregressive attention reuse
GQA: Grouped-Query Attention (Llama 3 uses
    8 KV-heads for 32 Q-heads)

Ridge Points (from our architectures):
    H100-class:     989 / 3350 = 0.295 FLOP/byte
    Inference-ASIC: 400 / 15000 = 0.027 FLOP/byte
    Unified L40S:   733 / 864 = 0.848 FLOP/byte

- 5. Related Work — Reference Implementations
- Training systems optimize parallelism and memory capacity across many GPUs. Inference systems optimize memory bandwidth, quantization, and dynamic scheduling on single chips. These are fundamentally different optimization targets — confirming the need for different hardware.

Slide 9:
- 7. FLOP Counting Model — 6P Training, 2P Inference
- Training FLOPs per token:  6 × P  =  6 × 8.03B  =  48.18 × 10⁹  (forward + backward + optimizer)
Inference FLOPs per token: 2 × P  =  2 × 8.03B  =  16.06 × 10⁹  (forward pass only)
P = total_parameters computed from Llama 3 8B config: H=4096, intermediate=14336, L=32, heads=32, KV-heads=8, vocab=128256
- Per-token FLOPs from project8_analysis.py:
- Training reads are low per-token because the large batch (B=4096) amortizes weight reads across many tokens
Inference reads are dominated by weight loading: every token must read all 8B INT8 parameters from memory
The 53× gap in arithmetic intensity (1,103 vs 20.85) is the quantitative proof that these are different workloads

Slide 10:
- 8. Memory Traffic Model
- Training Memory Traffic (per token)
- Weights: P × (precision_bytes + optimizer_bytes) / B
    precision_bytes = 2 (BF16), optimizer_bytes = 8 (Adam FP32 ×3)
    Amortized: 8.03B × 10 / 4096 ≈ 19.6 MB/token
Activations: 34 × H × L × S / B
    34 × 4096 × 32 × 4096 / 4096 ≈ 9.6 MB/token

Total read: 24.06 MB/token | Total write: 19.61 MB/token
- Inference Memory Traffic (per token)
- Weight loading: P × precision_bytes / B
    8.03B × 1 (INT8) / 16 = 501.8 MB/token
KV-cache read: 2 × L × (H × KV_heads/Q_heads) × S × bytes
    2 × 32 × 1024 × 4096 × 1 = 268.4 MB/token
KV-cache write: 2 × L × head_dim × bytes
    Negligible: 0.07 MB/token

Total read: 770.33 MB/token | Total write: 0.07 MB/token
- Key Insight: Training traffic is amortized by large batch → low per-token bytes. Inference traffic is dominated by weight loading and KV-cache reads → high per-token bytes.
- This is why inference is memory-bound on ALL architectures in our results.

Slide 11:
- 9. Bottleneck Model — Roofline Evaluation
- For each (architecture, workload) pair:

compute_bound_toks/s = peak_flops / flops_per_token
mem_bound_toks/s = bandwidth / bytes_per_token
effective = min(compute_bound, mem_bound)

If compute_bound < mem_bound → compute-bound
If mem_bound < compute_bound → memory-bound

Energy model:
    power = TDP × (0.4 + 0.6 × utilization)
    energy_per_token = power / throughput

EDP = energy_per_token × (1 / throughput)
- Example: H100 + Inference
- FLOPs per token: 16.06 × 10⁹
Peak: 989 TFLOP/s
compute_bound = 989e12 / 16.06e9 = 61,580 tok/s

Bytes per token: 770.33 + 0.07 = 770.40 MB
Bandwidth: 3,350 GB/s
mem_bound = 3350e9 / 770.4e6 = 4,348 tok/s

effective = min(61,580, 4,348) = 4,348 tok/s
    → MEMORY-BOUND

Compute utilization = 4,348 / 61,580 = 7.1%
    → 92.9% of H100 compute is IDLE

Slide 15:
- 11. Energy-Delay Product (EDP) Analysis
- EDP = Energy × Delay — jointly penalizes both high energy and high latency. Lower is better.
- Inference-ASIC achieves 185× better EDP than H100 for inference — driven by 42× lower energy AND 4.5× higher throughput
Unified-GPU is 7.5× WORSE than H100 on inference EDP — GDDR6 bandwidth bottleneck increases both latency and energy
For training, EDP values are comparable across architectures (1.09–1.66 µJ·s) — all are compute-bound and efficient

Slide 16:
- 11. TCO Analysis — Cost per Million Tokens
- 3-year TCO = chip_cost + electricity (TDP × 26,280 hours × $0.10/kWh). Cost/1M tokens = TCO / total_tokens.
- Inference Cost Comparison
- Inference-ASIC: $0.0017/1M tok
H100-class:     $0.0774/1M tok → 45× more expensive
Unified L40S:   $0.1029/1M tok → 61× more expensive

ASIC advantage: 10× cheaper chip + 9.3× less power

Slide 17:
- 12. Why a Unified Architecture Fails
- Unified-GPU (L40S-class) results from our framework demonstrate the irreducible conflicts
- Unified-GPU is worst for inference: 278.27 µJ·s EDP — 7.5× worse than H100, 1,391× worse than ASIC
Unified-GPU is mediocre for training: 15,213 tok/s — 26% slower than H100 (20,527 tok/s)
A unified chip carries the silicon area, power, and cost of both workloads but serves neither well
Bottom line: the compromise satisfies nobody — specialize or overpay

Slide 18:
- 13. Conclusion — Five Key Findings
- 1.  Training and inference are fundamentally different workloads.     AI gap: 1,103 vs. 20.85 ops/byte — training is compute-bound; inference is memory-bound.
2.  Training-optimized hardware wastes 93% of compute during inference.     H100: 61,580 tok/s compute capacity but only 4,348 tok/s effective (memory-limited).
3.  Specialized inference hardware delivers massive efficiency gains.     Inference-ASIC: 42× less energy/tok, 185× better EDP, 45× lower cost/1M tokens vs. H100.
4.  Unified/compromise architectures are the worst option for inference.     L40S-class: 7.5× worse EDP than H100, 1,391× worse than ASIC. GDDR6 bandwidth is fatal.
5.  The economic case for specialization is decisive.     As inference dominates production compute, purpose-built inference HW is a necessity.
